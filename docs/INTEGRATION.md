# RobustIDPS.ai integration

FedLoRAGuard is shipped as a dedicated *LoRA-adapter integrity* module inside
the RobustIDPS.ai platform.  The integration mirrors the architecture
described in §4.4 of the manuscript: each adapter ingested by the platform is
forwarded to the FedLoRAGuard scan service, and the verdict tuple
``(p_mal, k*, eps_T)`` is consumed by the platform's MITRE ATT&CK Mapper and
OWASP LLM Top 10 Scorecard.

## Compose

In the parent `robustidps_web_app/docker-compose.yml`, add:

```yaml
include:
  - ./FedLoRAGuard/docker-compose.yml

services:
  backend:
    environment:
      FEDLORAGUARD_URL: http://fedloraguard-svc:8000/scan_adapter
    depends_on:
      - fedloraguard-svc
    networks:
      - robustidps
```

## Backend client snippet

In `robustidps_web_app/backend/app/services/lora_integrity.py`:

```python
import os
import requests

ENDPOINT = os.environ["FEDLORAGUARD_URL"]


def scan_adapter(payload: dict) -> dict:
    resp = requests.post(ENDPOINT, json=payload, timeout=2)
    resp.raise_for_status()
    return resp.json()
```

## Verdict consumption

The platform's MITRE ATT&CK Mapper consumes ``mitre_attck_techniques`` from
the response.  The OWASP LLM Top 10 Scorecard consumes ``owasp_llm_top10``.
The ``certificate.k_star`` and ``certificate.epsilon_T`` are recorded in the
NIST AI RMF "Manage" tab of the compliance dashboard.

## RL active-investigation hook

When ``p_malicious`` falls between 0.4 and 0.7, the backend routes the
adapter to the DQN-based investigator (`fedloraguard.investigation.DQNInvestigator`)
which selects one of six fuzzing actions in a sandboxed environment.  The
action and reward are persisted in PostgreSQL for offline analysis.

## End-to-end smoke

```

curl -X POST http://localhost:8000/scan_adapter \
     -H "Content-Type: application/json" \
     -d @docs/examples/adapter.json
```
